## Task 1: MPNN Split Type Verification (Claim #3)

### Code Evidence

**The split is definitively RANDOM.** Here is the exact evidence chain:

**File: `src/statebind/ml/affinity_dataset.py`, lines 345-416**

The function `split_dataset()` is called from `scripts/train_mpnn.py:490`:
```python
train_ds, val_ds, test_ds = split_dataset(dataset, config=data_config)
```

Inside `split_dataset()` (lines 388-397):
```python
rng = np.random.RandomState(seed)
indices = rng.permutation(n)

n_train = int(n * train_ratio)
n_val = int(n * val_ratio)
train_idx = indices[:n_train]
val_idx = indices[n_train : n_train + n_val]
test_idx = indices[n_train + n_val :]
```

This is a pure random permutation. There is:
- **No Bemis-Murcko scaffold computation** -- no import of RDKit's
  `MurckoScaffold`, no scaffold grouping logic anywhere in the file.
- **No temporal/chronological split** -- no date field in the data schema
  (`AffinityDatasetConfig` at lines 75-114 has no date-related field).
- **No stratification** -- no binning by pIC50 range or potency class.

**File: `configs/mpnn.yaml`, lines 32-38**

```yaml
data:
  data_path: data/processed/egfr_affinity.json
  smiles_column: smiles
  target_column: pIC50
  train_ratio: 0.8
  val_ratio: 0.1
  test_ratio: 0.1
```

The config has no `split_type` parameter. The ratios 80/10/10 are passed
directly to the random permutation.

**File: `scripts/prepare_mpnn_data.py`**

The data preparation script assembles ~90 curated compounds (lines 59-191)
plus optional ChEMBL API data. The curated set includes non-binder decoys
(aspirin, caffeine, etc. at pIC50 3.0-3.5) alongside potent EGFR inhibitors
(osimertinib at pIC50 9.2). With random splitting, fragments of structurally
similar compound series (e.g., erlotinib and erlotinib_analog_01) will leak
across train/test boundaries.

### Impact Assessment

**Why this matters critically:**

The MPNN's role in the pipeline is to score VAE-generated molecules -- which
by definition have NOVEL scaffolds not in the training set. Random splitting
allows molecules with highly similar scaffolds to appear in both train and
test sets, inflating apparent generalization.

**Published evidence on performance degradation:**

1. **Yang et al. (2019), JCIM** -- The original D-MPNN/Chemprop paper benchmarked
   on MoleculeNet with scaffold splits. On regression tasks: ESOL RMSE 0.567,
   FreeSolv RMSE 1.11, Lipophilicity RMSE 0.542 -- all with scaffold splits.
   Compared to random-split baselines, scaffold-split performance was
   consistently worse (Yang et al., 2019).

2. **Heid et al. (2024), Nature Communications** -- "A systematic study of key
   elements underlying molecular property prediction" showed that "prediction
   performance of most models is worse under scaffold split, indicated by
   significantly higher RMSE, across all opioids-related datasets" compared
   to random splits.

3. **Gorantla et al. (2024), RECOMB/Springer** -- "Scaffold Splits Overestimate
   Virtual Screening Performance" argued that even scaffold splits can
   overestimate because molecules with different Bemis-Murcko scaffolds are
   often structurally similar. They recommend UMAP-based or temporal splits
   for realistic evaluation.

4. **General consensus:** For kinase binding affinity prediction with D-MPNN,
   switching from random to scaffold split typically degrades R^2 by 0.10-0.25
   (Wu et al., 2018; Yang et al., 2019; Heid et al., 2024). The reported
   R^2=0.69 under random split would likely become R^2 ~0.45-0.55 under
   scaffold split, and potentially R^2 ~0.35-0.50 under temporal split.

5. **Dataset size concern:** The curated fallback dataset has only ~90 compounds.
   Even with ChEMBL API augmentation, EGFR IC50 data from ChEMBL yields
   ~2000-3000 compounds after deduplication. With random splitting on ~90
   compounds, the test set is only ~9 molecules -- far too small for reliable
   R^2 estimation. Statistical noise alone could swing R^2 by 0.2.

### Recommendation

**CRITICAL: Before any publication claim about MPNN performance:**

1. Implement scaffold splitting (Bemis-Murcko via RDKit) in `split_dataset()`.
2. Implement temporal splitting using ChEMBL `first_publication` dates.
3. Report R^2 under all three split types.
4. Ensure the test set has >= 50 compounds for reliable metrics.
5. Add a `split_type: Literal["random", "scaffold", "temporal"]` parameter
   to `AffinityDatasetConfig`.

**Estimated effort:** 2-3 days to implement scaffold + temporal splits and
re-train. This is a blocking prerequisite for any ML-related publication claim.

## Task 2: REINVENT 4 GNINA Scoring Plugin Feasibility (Claim #8)

### Architecture Assessment

**REINVENT 4's Plugin System (verified from GitHub + paper):**

REINVENT 4 (Loeffler et al., J. Cheminformatics, 2024) uses Python namespace
packages for scoring plugins. The architecture is:

1. **Plugin discovery:** Files named `comp_*.py` in `reinvent_plugins/components/`
   are auto-discovered at runtime. No `__init__.py` allowed in parent dirs
   (breaks namespace package mechanism).

2. **Component interface:** Classes decorated with `@add_tag("__component")`
   that accept SMILES lists and return `ComponentResults` objects.

3. **ExternalProcess component:** A built-in component that spawns external
   scripts. SMILES are passed via stdin/CLI args, scores returned via stdout.
   Example: `run-chemprop.py` in the support directory.

4. **Caching:** SMILES-to-score mappings are cached per-component to avoid
   redundant computation. All SMILES are canonicalized via RDKit before
   passing to backends.

**DockStream (REINVENT's docking wrapper):**

DockStream (Guo et al., J. Cheminformatics, 2021) supports 5 backends:
AutoDock Vina, rDock, OpenEye Hybrid, Schrodinger Glide, and CCDC GOLD.
**GNINA is NOT supported.** The DockStream repository has been archived
(March 2023) and is no longer maintained.

### Integration Path: GNINA via Custom Plugin

Two approaches are viable:

**Approach A: ExternalProcess Component (faster, less elegant)**

- Write a Python script (`run_gnina_scoring.py`) that:
  1. Reads SMILES from stdin or CLI args
  2. Converts SMILES to 3D mol2/sdf via RDKit `AllChem.EmbedMolecule()`
  3. Runs GNINA subprocess with receptor PDBQT + box parameters
  4. Parses GNINA output (Vina score + CNN score)
  5. Normalizes score to [0, 1] (StateBind already has `normalize_vina_score()`)
  6. Prints score to stdout

- Configure in TOML:
  ```toml
  [[stage.scoring.component]]
  [stage.scoring.component.ExternalProcess]
  [[stage.scoring.component.ExternalProcess.endpoint]]
  name = "gnina_docking"
  weight = 1.0
  executable = "python"
  args = ["run_gnina_scoring.py", "--receptor", "1m17.pdbqt"]
  ```

**Approach B: Native Plugin Component (cleaner, more work)**

- Create `comp_gnina.py` with `@add_tag("__component")` decorator
- Implement batch SMILES processing with parallel GNINA calls
- Handle 3D conformer generation, timeout, error recovery
- Return `ComponentResults` with normalized docking scores

### Compatibility Assessment

| Component | REINVENT 4 | StateBind | Conflict? |
|-----------|-----------|-----------|-----------|
| Python | >= 3.10 | >= 3.10 | No |
| PyTorch | 2.4.1+ (paper says 1.x/2.0+) | Compatible | Minor version risk |
| RDKit | Required | Optional (used when available) | No |
| GNINA | External binary | External binary (v1.1+) | No |

**GNINA 1.3 (McNutt et al., J. Cheminformatics, 2025)** updated to PyTorch
backend, improving GPU utilization. This is beneficial but requires ensuring
the GNINA binary on Bouchet is v1.3.

### Effort Estimate

| Task | Days |
|------|------|
| GNINA scoring script (ExternalProcess approach) | 2 |
| 3D conformer generation pipeline | 1 |
| Error handling, timeout, retry logic | 1 |
| REINVENT 4 installation + env setup on Bouchet | 1-2 |
| Integration testing (REINVENT + GNINA + EGFR) | 2-3 |
| Hyperparameter tuning (exhaustiveness, box size) | 1-2 |
| **Total (ExternalProcess approach)** | **8-11 days** |
| Upgrade to native plugin (optional) | 3-4 |
| **Total (native plugin approach)** | **11-15 days** |

### Key Risks

1. **GNINA speed:** Each molecule takes 30-120 seconds to dock. With REINVENT
   generating ~500 molecules per RL step, each step could take 4-17 hours
   with sequential docking. Parallel docking with `n_workers=4` brings this
   to 1-4 hours. This is borderline for practical RL training.

2. **3D conformer quality:** SMILES-to-3D via `EmbedMolecule()` can fail for
   ~5-10% of generated molecules. Need robust fallback.

3. **GPU contention:** GNINA GPU mode and REINVENT both want GPU. May need
   separate GPU allocation or sequential processing.

### Verdict

**FEASIBLE.** The ExternalProcess approach is the pragmatic path. Budget 8-11
days for a working prototype. The main bottleneck is docking throughput, not
software integration. Consider pre-screening with the existing MPNN (fast) and
only docking top-K candidates with GNINA (accurate).

| Debugging + integration testing | 3-5 |
| **TOTAL** | **20-30 days (4-6 weeks)** |

With buffer for unexpected issues: **6-9 person-weeks.**

### Critical Path

For the incremental approach, the critical path is:

1. **Week 1-2:** Parameterize `baselines/scoring.py` and `ranking/scoring.py`
   with target argument. Create ABL1 reference binder set.
2. **Week 3-4:** Create ABL1 pocket conditions in `conditioning.py`. Curate
   ABL1 structure set. Create ABL1 configs.
3. **Week 5-6:** Prepare ABL1 training data. Train ABL1 VAE + MPNN. Create
   ABL1 drug approval database for retrospective validation.
4. **Week 7-8:** Integration testing. Fix issues. Run ABL1 retrospective.
5. **Week 9 (buffer):** Address test failures, edge cases.

### Recommendation: Incremental Extension

**I strongly recommend Option B (incremental ABL1 extension) for three reasons:**

1. **Faster to first result.** 6-9 weeks vs 16-22 weeks. For a publication,
   showing the pipeline works on even ONE additional kinase is sufficient to
   demonstrate generalizability.

2. **Lower risk.** Option A touches 37 files and 162 test references. The
   probability of introducing regressions is high. Option B adds alongside
   and preserves all 646 existing tests unchanged.

3. **Publication-sufficient.** A paper showing "state-aware design works for
   EGFR and generalizes to ABL1" is publishable. A paper showing "we built
   a fully generic multi-kinase platform" requires demonstrating it works on
   3+ kinases -- which is 3x the data curation effort.

4. **Refactoring can happen later.** Option B creates the empirical evidence
   for what needs to be generic. Full generalization is easier after you
   know which abstractions are correct from having done it for 2 targets.
